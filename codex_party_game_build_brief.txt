PROJECT BRIEF FOR CODEX
Title: Online Multiplayer Party Game Platform
Version: MVP v1
Audience: Codex / coding agent
Owner context: Solo student builder with limited budget
Primary goal: Build a playable, deployable web platform for hosting online multiplayer party-style games with guest users, registered accounts, custom question packs, live room gameplay, and optional voice/video support.

================================================================================
0. CORE INSTRUCTION TO CODEX
================================================================================

You are building a real web product, not a demo toy.

Your job is to create a production-structured MVP for an online multiplayer party game platform that supports:
- guest users
- registered users
- host-created private rooms
- join-by-room-code flow
- saved custom question packs for registered users
- live multiplayer gameplay
- multiple reveal/play styles
- text chat
- microphone voice chat
- optional camera/video architecture hooks

You must optimize for:
- low hosting cost
- student-friendly deployment
- fast MVP delivery
- clean future scalability
- maintainable code
- strong TypeScript typing
- basic security and abuse prevention

Do NOT overbuild the first version.
Do NOT add unnecessary microservices.
Do NOT implement 8 different games immediately.
Do NOT build complex matchmaking.
Do NOT build payments.
Do NOT build a native mobile app.

Build the MVP first, but structure the code so new game modes can be added later.

You must generate:
1. a monorepo or clean multi-app repo structure
2. frontend app
3. backend/game room state service
4. database schema and migrations
5. auth integration
6. live room state management
7. one working end-to-end playable game mode

================================================================================
1. PRODUCT VISION
================================================================================

This platform is a web-based online multiplayer social party game system.

Users can:
- play as guests without full signup
- create accounts and save their own content
- host private rooms for friends
- join rooms with room codes
- play party games using prompts/questions
- answer prompts
- reveal answers either all at once or one at a time
- vote, score, and continue through rounds
- use voice chat through microphone
- possibly use camera later
- create and save custom question packs
- host using their own saved packs

The feel should be:
- easy to join
- fast to start
- fun in a group
- smooth for hosts
- mobile-friendly enough in browser
- modern but not overly flashy

Think of the product as:
“an online multiplayer party game platform for friends, with room-based play, custom prompts, and flexible game modes.”

================================================================================
2. MVP SCOPE
================================================================================

The MVP must include ONLY the essential features below.

2.1 User types
- Guest user:
  - can join rooms
  - can play games
  - can chat
  - can use voice in room
  - cannot permanently save private question packs unless upgraded
- Registered user:
  - can sign up / log in
  - can create rooms
  - can save and manage personal question packs
  - can host games using saved packs
  - can view basic game history later if implemented cleanly

2.2 Room system
- create private room
- generate room code
- join room by code
- lobby before game start
- player list
- host controls
- ready/unready state
- start game button for host
- lock room option
- kick player option
- simple reconnect flow if a browser refresh happens

2.3 Gameplay
Implement at least 1 complete mode, but architect for more.

Required supported mechanics in engine:
- prompt shown to players
- answer submission phase
- submission lock after timer or host action
- reveal phase
- reveal style A: all answers shown at once
- reveal style B: one answer shown at a time
- scoring
- round progression
- game end summary

2.4 Content
- built-in default prompt pack
- ability for registered users to create custom question packs
- ability to edit/delete their own packs
- ability to choose a pack when hosting

2.5 Communication
- text chat in room
- voice chat with microphone support
- camera/video not required in MVP UI, but architecture should leave room for later support

2.6 Basic moderation / safety
- host can mute text chat for room if needed
- host can kick players
- host can lock room
- simple rate limiting
- profanity filtering hook for usernames and pack names
- room expiration

2.7 Out of scope for MVP
- public matchmaking
- payments/subscriptions
- friend graph/social network
- achievements
- cosmetics
- tournaments
- advanced moderation dashboard
- deep analytics dashboards
- native apps
- full video support in first release
- many different game genres at launch

3. RECOMMENDED TECH STACK
================================================================================

Use the following stack unless there is a strong implementation reason to deviate.

3.1 Frontend
- Next.js
- TypeScript
- App Router preferred
- Tailwind CSS
- Zustand or equivalent light state solution if needed
- React Query / TanStack Query for API data fetching if useful

3.2 Backend / persistent platform services
- Supabase for:
  - Postgres
  - Auth
  - Storage
  - possibly some realtime/presence where appropriate

3.3 Real-time room/game state
- Cloudflare Workers + Durable Objects OR equivalent authoritative room service
- The active room/game state must be server-authoritative, not client-authoritative

3.4 Voice/video
- Daily SDK OR LiveKit
- Start with microphone/audio only in UI
- Keep camera/video optional and disabled by default
- Build abstraction layer so provider can be swapped later

3.5 Hosting
Preferred target architecture:
- frontend deployed to Vercel or Cloudflare Pages
- game room backend deployed to Cloudflare Workers
- database/auth on Supabase
- audio/video provider externalized

3.6 Repo / package management
- pnpm monorepo preferred
- use shared packages for types and game engine logic

================================================================================
4. HIGH-LEVEL ARCHITECTURE
================================================================================

Use a clear separation of concerns.

4.1 Persistent vs live state
Persistent data belongs in database:
- users
- profiles
- question packs
- prompts
- room metadata
- past game sessions
- analytics events
- moderation logs

Live room state belongs in authoritative room service:
- connected players
- ready states
- current round
- current timer
- submitted answers in active round
- reveal order
- current scores
- host commands
- transient chat events if not persisted

Do NOT rely on the frontend as the source of truth for game state.
Do NOT rely on client timers for critical round logic.
Do NOT use only database polling for gameplay.

4.2 Suggested repo structure
Create something close to this:

/apps
  /web
  /room-server

/packages
  /ui
  /types
  /game-engine
  /db
  /shared
  /voice

/docs
  product-spec.md
  architecture.md
  local-development.md
  deployment.md

/supabase
  /migrations
  /seed

/infrastructure
  env.example
  cloudflare
  vercel

4.3 Responsibilities
apps/web:
- landing page
- login/signup/guest join
- dashboard
- pack editor
- room lobby
- live room UI
- host controls
- chat panel
- voice controls

apps/room-server:
- room creation coordination
- room join validation
- authoritative game state
- websocket handling
- round state machine
- timers
- scoring
- server-side validation of gameplay actions

packages/game-engine:
- generic game state interfaces
- mode config
- phase transitions
- scoring logic contracts
- reusable utilities

packages/types:
- shared TypeScript types
- event contracts
- DTOs
- API payload types

packages/db:
- typed database access helpers
- schema adapters if useful

packages/voice:
- abstraction for Daily/LiveKit provider integration

5. GAME ENGINE DESIGN
================================================================================

The platform must be designed as a game engine + configurable modes, not as one giant hardcoded game.

5.1 Core gameplay concepts
Design around these concepts:
- Room
- Player
- Host
- GameSession
- Round
- Prompt
- Submission
- RevealPhase
- VotePhase
- ScoreRule
- TimerRule
- WinRule
- GameMode

5.2 Game mode model
A mode should be configurable with fields such as:
- id
- name
- description
- answer style
- reveal style
- timer duration
- whether voting is enabled
- how scoring works
- round count
- whether host moderates reveal progression

Example ideas:
- answerStyle: simultaneous / turn_based
- revealStyle: all_at_once / one_by_one
- scoringRule: audience_vote / host_choice / custom

5.3 MVP mode requirement
Implement one complete working mode using the engine.
Example MVP mode:
- everyone gets the same prompt
- everyone submits one answer
- answers are locked
- answers are revealed all at once or one at a time depending on room settings
- players vote for favorite answer except their own
- scores update
- next round starts
- after N rounds, final scoreboard appears

5.4 Future-proofing
The code should make it easy later to add:
- hidden role modes
- judge-based modes
- team modes
- bluffing modes
- rotating prompt ownership
- host-driven custom scoring

================================================================================
6. USER FLOWS
================================================================================

Implement these flows cleanly.

6.1 Guest flow
- user lands on homepage
- user clicks “Join as guest”
- user enters display name
- temporary guest profile/session created
- user enters room code or joins via invite link
- user waits in lobby
- user plays game
- on exit, data is temporary
- optional prompt to upgrade to account later

6.2 Registered user flow
- sign up / log in
- create profile
- create/edit/delete question packs
- create room
- configure room settings
- choose game mode
- choose question pack
- invite friends with room code or URL
- host game
- optionally resume account session later

6.3 Host flow
- create room
- room code generated
- choose settings:
  - max players
  - reveal style
  - prompt pack
  - round count
  - timer length
  - room lock
- players join
- host starts game
- host can:
  - kick player
  - lock room
  - end game
  - advance phase if needed
  - mute chat if implemented

6.4 Reconnect flow
- if player refreshes page, attempt rejoin using session token
- restore seat if still valid
- restore current room state
- prevent duplicate player instances where possible

8.1 Auth requirements

- guest access without forced signup

- protected dashboard routes

9. REAL-TIME ROOM SERVER
================================================================================

This is critical.

9.1 Rule
The room server is the authority for active room state.

9.2 Responsibilities
- create room instances
- track connected sockets
- manage lobby state
- manage ready state
- manage game phase transitions
- handle timers
- validate actions
- broadcast state updates
- recover or rebuild state from persistence if needed
- expire abandoned rooms

9.3 Connection model
Use WebSockets for room state synchronization.

9.4 Events
Define typed client-to-server and server-to-client events.

Client-to-server examples:
- join_room
- leave_room
- ready_up
- unready
- start_game
- submit_answer
- vote_submission
- reveal_next
- send_chat_message
- kick_player
- lock_room
- end_game
- reconnect_room

Server-to-client examples:
- room_state
- room_joined
- player_joined
- player_left
- player_ready_changed
- game_started
- round_started
- timer_started
- submissions_locked
- reveal_state_changed
- chat_message
- scores_updated
- game_ended
- room_error

9.5 Validation
Server must validate:
- room membership
- phase correctness
- duplicate actions
- timing rules
- host-only actions
- answer length/content rules
- vote validity

20. DEFINITION OF DONE FOR MVP
================================================================================

The MVP is done when all of the following are true:

- a guest can join a room
- a registered user can create a room
- a registered user can create and save a question pack
- host can choose a pack and start a game
- multiple players can join and see live synchronized state
- all players can submit answers
- answers can be revealed all at once or one at a time
- votes and scores work
- text chat works
- voice chat works for microphone audio
- reconnect after refresh works reasonably
- deployment instructions are complete
- local setup instructions are complete
- core flows have tests
- app is structured cleanly enough for future expansion

================================================================================
21. IMPORTANT DO-NOT RULES
================================================================================

Do NOT:
- keep all gameplay logic in React components
- use client as source of truth for scores/timers
- hardcode one giant all-purpose game component
- scatter event names without shared types
- expose secrets to the browser
- let guests edit saved packs that belong to accounts
- build many game modes before one works end-to-end
- implement full camera/video before audio is stable
- skip docs
- skip tests for the game engine
